Changes in general

 

·         Changed license to Apache License 2.0
Many have asked why WatiN uses the LGPL. People where anxious to use WatiN with there commercial applications because of all the uncertainties regarding the use of *GPL and forcing this license upon every application that uses *GPL licensed software. The reason I chose the LGPL instead of the GPL license was because I thought it was meant to make it possible to WatiN with commercial licenses. It was never meant to rule out anyone.

After a lot of thinking, reading and discussing licenses with the community, I decide to change the WatiN license to the (much) more open Apache License 2.0. Visit http://www.apache.org/licenses/LICENSE-2.0.html to read the full license text.

Of course I hope this change will lead to using WatiN more with commercial application. I also hope to see more contributions from these “commercial” users which will benefit everybody.

 

·         Added support for finding an element by multiple attributes
You can combining attributes everywhere you supply an Attribute class (or one of it's inheritors) to find or check for the existence of an element on a page, an Internet Explorer instance, a HTMLDialog or a Frame. Some examples:

 

// Finding a button by it's name and value

Button button = ie.Button(Find.ByName("button1").And(Find.ByValue("Cancel")));

 

// Finding a button by it's English or Dutch value

Button button = ie.Button(Find.ByValue("Annuleren").Or(Find.ByValue("Cancel")));

 

// Finding a button by it's name and it's English or Dutch value

Button button = ie.Button(Find.ByName("button1").And(Find.ByValue("Annuleren").Or(Find.ByValue("Cancel"))));

 

// The same but with operators && and || (works also with & and | but has the same effect as && or ||)

Button button = ie.Button(Find.ByName("button1") && (Find.ByValue("Annuleren") || Find.ByValue("Cancel")));

 

// Getting a filtered collection of buttons

ButtonCollection buttons = ie.Buttons.Filter(Find.ByValue("Annuleren") || Find.ByValue("Cancel"));

 

// Checking the existence of a button

bool exists = ie.Button(Find.ByValue("Annuleren") || Find.ByValue("Cancel")).Exists;

 

// Finding the second button by it's name

Button button = ie.Button(Find.ByIndex(1).And(Find.ByName("button1")));

 

// Finding the second Internet Explorer by it's title

IE ie = IE.AttachToIE(Find.ByIndex(1).And(Find.ByTitle("WatiN")));

 

As seen in the fourth example, you can also make use of the && and || operators. The & and | operators are also support but will always be evaluated as ConditionalAnd (&&) or ConditionalOr (||). So using & or && and | or || gives the same result. Since this behaviour might change in the future it’s recommended to only use the && and || operators.

 

With the introduction of this new feature, the IdAndOccurrence class is removed. If you used this class in your tests simply replace it with Find.ById() && new Index() like this:

 

Before WatiN 1.0 code

Button button = ie.Button(new IdAndOccurrence(“buttonId1”, 1));

 

With WatiN 1.0 new multiple attribute support this would be:

Button button = ie.Button(Find.ById(“buttonId1”) && Find.ByIndex(1));

 

 

Attribute class compatibility is broken with previous versions of WatiN:

Introducing this feature forced some changes to made in the Attribute class. This might affect those who have written their own subclasses of Attribute and did override one of the Compare methods.

 

The Compare(string) and Compare(object) methods are removed and Compare(IAttributeBag) is the replacement for these two methods. The recommended way to get an attribute value is to use the GetValue method on the IAttributeBag interface. If you did something special with the element object or did access properties of the element which didn’t return a simple type (string, int, etc.), you can cast the attributeBag argument to ElementAttributeBag and access the IHTMLElement property to get access to the element. Make sure you read the API documentation regarding the following methods of the Attribute class before you start changing your code: Compare, doCompare, LockCompare and UnlockCompare.

 

 

·         Added full support to test Style attributes of an element
With this new release it’s possible to get the values of the different attributes within the style attribute. Use the Element.Style property on each element type. This will return a Style object representing the actual values in the style attribute. The style attribute or one if it’s inner attributes can also be used to find elements on a webpage. Some examples:

 

// Get the color of the button text using the Color property of the Style object

Assert.AreEqual("white", ie.Button("buttonid1").Style.Color;

 

// Get the background color of the button using the GetAttributeValue method. The attribute name

// is written as the attribute name in the style html code: background-color

Assert.AreEqual("blue", ie.Button("buttonid1").Style.GetAttributeValue("background-color");

 

// Get the background of the button using the GetAttributeValue method. The attribute name

// is written as the property name of the DOM style object: BackgroundColor

Assert.AreEqual("blue", ie.Button("buttonid1").Style.GetAttributeValue("BackgroundColor");

 

// Find an element by the value of the background color

Assert.IsTrue(ie.Button(Find.ByStyle("background-color", "blue")).Exists);

 

// Get the whole style attribute text through Element.GetAttributeValue(“style”).

// Element.Style.CSSText will return the same result.

Assert.AreEqual("COLOR: white; BACKGROUND-COLOR: blue", ie.Button("buttonid1").GetAttributeValue("style");

 

// Get the style background-color attribute through Element.GetAttributeValue using the style.attributename syntax

Assert.AreEqual("blue", ie.Button("buttonid1").GetAttributeValue("style.background-color");

 

// Find an element by the whole style attribute value

Assert.IsTrue(ie.Button(Find.ByCustom("style", "COLOR: white; BACKGROUND-COLOR: blue")).Exists);

 

 

·         Cast an Element instance to a more specialised element type
With previous versions of WatiN it wasn’t possible to cast an Element return value, returned by a method or property, to a more specialised WatiN element type. With this release all properties and methods that have a return type Element will return a specialised element type. The only exception to this is the ie.Element(findBy) method, but all elements in the ElementCollection returned by the ie.Elements property are of a specialised type though. Following an example:

HTML snippet:

<div>

  <textarea id="id1" />

</div>

 

Here is how you get the typed parent div of this TextField:

 

Div div = (Div)ie.TextField("id1").Parent;

 

To make this possible all WatiN elements have a new constructor which accepts an Element as an argument. This can be used to create a specific element type from the return value of the ie.Element(findBy) method.

 

Element element = ie.Element("id1");

textField textfield = new TextField(element);

 

If the field element can’t be converted to a TextField, an ArgumentException is thrown.

 

Other changes that resulted from returning specialised types are:

o        Element.Parent will return null when element is the root element in the DOM hierarchy.

o        Element.NextSibling will return null when element has no next sibling.

o        Element.PreviousSibling will return null when element has no previous sibling.

 

 

·         Introducing the Settings class
You can now change many default settings used by WatiN.

 

// Increase the time WatiN waits for a page load to become complete (max 300 seconds)

IE.Settings.WaitForCompleteTimeOut = 300;

 

This is a list of the settings and their default values:

Setting

Default

Description

AttachToIETimeOut

30

Set the default time in seconds used when waiting for an Internet Explorer matching the given findBy.

WaitUntilExistsTimeOut

30

Set the default time in seconds WatiN will wait for an element to appear on a webpage.
Setting this value to zero will give you immediate feedback if en Element can not be found instead waiting the default period before throwing the ElementNotFoundException

WaitForCompleteTimeOut

30

Used by WatiN when it calls the WaitForComplete internally. So if you have a web loading after a button.Click which takes more then 30 seconds, you should increase this value to prevent a TimeOutException.

HighLightElement

True

Used by WatiN to determine if elements should be highlighted while running the tests.

HighLightColor

Yellow

If elements should be highlighted this is the highlight color.

AutoCloseDialogs

True

Should the dialogwatcher close unhandled dialogs automatically. Setting this property has only effect when done before any IE instance is referenced. You can set this after an IE instance is created through ie.DialogWatcher.CloseUnhandledDialogs = false;

 

 

·         Improved SelectList

With this release you can get full access to the Option elements within a SelectList. Among many other possibilities, the new implementation offers you an easy way to select an option by its index and to select an element and not wait for the page to complete loading. Example:

 

ie.SelectList(findBy).Options[4].SelectNoWait();

 

See the API documentation (watin.chm) for more info on the available properties and methods.

 

 

·         Improved Table support

New in this version is the addition of the TableBody element type and the TableBodyCollection. Thanks to William Yeung for sending in the patch for this! Some examples of using this new TableBody:

 

TableBody tbody = ie.TableBody(findBy);

TableBody secondtbody = ie.TableBodies[1];

 

The previous examples will return a TableBody not specifically related to any table. Table.TableBody(findBy) and Table.TableBodies, however, will get only the TableBodies belonging to that table and not from any nested tables within that table.

 

Same thing goes for the TableRows you acces from the TableBody class. TableBody.TableRow(findBy) and TableBody.TableRows will only return the table rows belonging to that TableBody and not from any nested tables within that table body.

 

Other changes are:

o        Added TableRow TableCell.ParentTableRow

o        Added int TableCell.CellIndex

o        Added Table TableRow.ParentTable

 

 

·         Added Element.WaitUntilRemoved to complete the WaitUntilExists method introduced in the previous release. WaitUntilRemoved waits until the element is actually removed from the page (by java script). Element.Exists will also return false if an element was found on a page but got removed after finding it. An example:

 

Button clickOnlyOnceButton = ie.Button("clickOnlyOnceButtonId")

clickOnlyOnceButton.ClickNoWait; // Java code will remove this button from the page

clickOnlyOnceButton.WaitUntilRemoved(15);

Assert.IsFalse(clickOnlyOnceButton.Exists);

 

·         Improved LogonDialog handler by setting the focus to the username field before entering username and responds better on PC’s under heavy load. Thanks to Peter McNulty for supplying the patches.

 

·         Improved FileUploadDialogHandler handler by setting the focus to the edit box where the path\file must be. In Vista the FileUpload dialog wasn’t recognised. Thanks to Jesus Garza for supplying the patch to fix this.

 

·         Added SecurityAlertDialogHandler. It will always click the OK button if an security alert pops up. Example:

 

ie.AddDialogHandler(new SecurityAlertDialogHandler());

 

·         Added FrameCollection.Exists(findBy)

 

·         Added HTMLDialogCollection.Exists(Attribute)

 

·         Added Element.FireEventNoWait(eventname) which fires the event but doesn't wait for IE to complete.

 

·         It is now possible to type text into a multiline textfield with a NewLine character in the text to type. In previous versions this resulted in two newlines showing up in the textfield.

 

·         Added constructor overload to Attribute which accepts string attributeName, ICompare comparer. This makes it possible to use your own comparer implementations when finding an Element. The following example assumes the existence of a StartsWithComparer class:

 

Link link = ie.Link(new Value(new StartsWithComparer(“Del”));

 

This is the first step in implementing support for an additional syntax to find elements, planned for the next release:

 

Link link = ie.Link(Value.StartsWith(“Del”) && Index.Equals(2));
